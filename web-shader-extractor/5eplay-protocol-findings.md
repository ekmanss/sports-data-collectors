## `web-shader-extractor` 适用性结论

Chrome 对赛程列表页和比赛详情页的渲染表面清点均未发现 Canvas、WebGL、WebGPU、视频、iframe 或 SVG 特效目标；页面是由 HTTP/MQTT 数据驱动的普通 React DOM。按照 `web-shader-extractor` 的目标锁定门槛，本任务在 `TARGET_LOCK_GATE` 处判定为“不适用着色器提取”，没有继续构造不存在的 Shader、纹理、渲染图或本地回放。范围证据记录在 `scout-card.json`、`run-state.json`、`known-gaps.md` 和 `evidence/runtime/surface-inventory-*.json`。后续协议调查独立使用 Chrome 页面、网络响应、目标路由源码和连续状态观察完成。

## 从路由实际加载源码确认的页面合并行为

- 拒绝 `match.mc_info.id` 与当前路由比赛 ID 不一致的 MQTT 状态更新。
- 对最近 20 个 `from_ver` 去重。
- 合并 MQTT 状态前，丢弃传入的 `global_state.t1_odds_percent` 和 `t2_odds_percent`。
- 按 `bout_num - 1` 作为数组下标替换每个传入地图状态。
- 对传入的 `global_state` 做浅合并。
- 只有 `info.match_id` 等于当前路由比赛 ID 时，才接受实时事件更新。

在一段正常增量过程中，实际观察的状态主题版本链通常连续：下一条非零 `from_ver` 等于上一条增量消息的 `this_ver`，定期 HTTP 的 `state_ver` 也通常与增量 MQTT 版本对齐。但它不是全局单调的单链。地图开局或边界可能发送 `from_ver` 全零的普通 `csgo-detail` 包。例如，图 2 结束消息为 `from_ver=0001784487374, this_ver=0001784487390`；1.124 秒后又收到 `from_ver=0000000000000, this_ver=0001784487392` 的 `csgo-detail`，其中包含完整的已结束图 1 对象；图 3 开始时，增量包继续使用 `from_ver=0001784487390`，而不是 `...7392`。因此，全零包属于并行的基线/重置分支，不能推进普通增量链游标。应对其做幂等合并并与 HTTP 对账，不能把它当作普通断链。

`csgo_mc_2396047` 从赛前切到进行中时，Chrome 在详情主题收到 `event_name: "csgo-detail-bp"`，其中 `global_state.status: "1"`。页面分发器没有处理该事件名，导致已打开标签页停留在过时的赛前界面。刷新后，页面从 HTTP 快照立即渲染 `LIVE 0-0`、实时日志区、地图禁选和三张未开地图。可靠采集器在等待图 1 时必须持续轮询 `/data`，不能只依赖页面能够识别的两种 MQTT 事件名。

2026-07-20 下午的两个新样本进一步限定了 `csgo-detail-bp` 的语义：`csgo_mc_2395919` 和 `csgo_mc_2395546` 分别在 15:58:34、15:58:45 收到首条 BP 消息，但随后 HTTP 仍为 `global=0, [-1,-1,-1]`，只是三张地图名称已经填入；直到 16:01:55、16:02:05，HTTP 才分别切到 `global=1, [-1,-1,-1]`。因此，BP 只能视为“地图禁选或比赛基线已变化，需要立即重拉 HTTP”的失效通知，不能直接判为比赛已经开始。两个已打开的 Chrome 页面在 HTTP 已变为 `global=1` 后仍停留在赛前分析界面；重载后才显示 `LIVE 0-0`、完整地图 BP 和三张 `-:-` 地图卡片。

计划时间继续改到 02:20 和 02:35 后，同一场再次进入“系列赛进行中/图 1 未开”。观察器在 02:36:07.969 和 02:36:50.274 又收到两条 `csgo-detail-bp`，两者均为 `from_ver: "0"`、`global_state.status: "1"`。已打开页面仍停留在旧赛前界面，甚至显示过时的 02:05 时间。实际刷新标签页后，页面才从 HTTP 显示 `LIVE 0-0`、地图禁选、实时日志和三张未开地图。重复出现相同行为，排除了偶发丢消息的解释。

事件历史接口还以 `limit=10` 做过翻页实测：`update_version=0` 返回按 `info.update_version` 从新到旧排列的数据；将当前页最后一条（最旧一条）的 `update_version` 作为下一页参数，会得到更旧数据且不重复游标行。响应在确实仍有更旧数据时仍返回 `not_more: "1"`，所以完整采集器不能把该字段当作历史已完整的标志。应在空页、短页或游标重复时停止，并按事件 `update_version` 去重。

`csgo_mc_2395920` 还暴露了技术结果边界：图 1 Ancient 正常以 13:6 结束后，HTTP 从 `global=1, [2,-1,-1]` 直接跳到 `global=2, [2,2,2]`；图 2 Cache 被记为 1:0，但没有开始/结束时间，图 3 也被标为 `status=2` 却没有比分和结果。重载后的 Chrome 页面显示 2:0、Cache 1:0、图 3 `-:-`。因此，`bout.status=2` 只能表示该地图槽位已结算/关闭，不能单独证明地图实际打过；判断“实际进行过”还必须检查 `start_time`、比分、`result` 和事件历史。

补充规则：

- `live_status` 在全部直接观察状态中始终是 `0`，禁止用它分类。
- `plan_ts` 只用于调度，不用于状态判断；它会改期、越时。
- 事件日志可在正式开图前活跃，不能用“有击杀/有第 1 回合日志”代替 `bout.status=1`。
- 图 1 尚未开始时，允许 `live_map1_unopened → prestart → live_map1_unopened`。`csgo_mc_2395920` 在 19:17:13、19:19:18、19:20:23 直接复现了这条回退/重进链。
- `complete` 只表示系列赛结束；每张图另输出 `played = start_time非空`，并用比分、`result` 与事件历史做一致性校验。`status=2` 但无 `start_time` 的技术结算槽位不得视作实际开图；若字段互相矛盾，则暂记未知并重同步。
- 如果 `global=2` 但仍有 `bout.status=1`，或地图序列出现“后图已开而前图未结算”等矛盾，返回 `unknown_inconsistent`，不得对外确认状态。

## 推荐的代码采集方案（设计，不是产品实现）

最稳妥的方案不是模拟浏览器抓 DOM，而是复刻页面的数据合并语义：以 HTTP `/data` 为权威基线，以 MQTT 降低实时延迟，以定期 HTTP 对账修复遗漏和回滚；DOM 只用于调查/验收，不作为生产数据源。

### 1. 三层数据模型

1. **原始证据层**：不可变保存每次 HTTP 响应和 MQTT 信封，记录 `captured_at`、来源、主题、比赛 ID、版本和 SHA-256；不得保存 MQTT 临时凭据或账号会话材料。
2. **页面等价规范层**：严格按页面规则合并 `/data` 与 `csgo-detail`：验证比赛 ID、浅合并 `global_state`、按 `bout_num-1` 替换地图对象，并忽略实时包中的两个赔率百分比字段。
3. **类型化投影层**：从规范层输出稳定接口，例如 `phase`、系列赛比分、每图 `settled/played/live`、选手状态、BP、事件时间线和数据新鲜度；业务代码不要直接依赖原始 JSON 的偶发缺字段。

### 2. 启动顺序

1. 校验比赛 ID，并构造详情主题 `csgo/product/detail/{id}` 与事件主题 `csgo/product/event/log/{id}`。
2. 调用 `/api/restrict/matchscore` 获取短期 MQTT 连接材料，只保存在内存。
3. **先订阅 MQTT 并缓冲消息，再请求 HTTP `/data` 基线**，避免“拉基线期间漏掉边界”的竞态。
4. 获取 `/data` 后建立规范状态，再按版本顺序回放缓冲消息；无法证明连续时直接重新拉 `/data`，不猜缺失更新。
5. 并行获取页面静态/补充数据：`analysis_v1`、两套战队近期比赛接口、社区评分接口和聊天室可用性。只有完赛后再请求赛事内容/集锦接口。
6. 首次加载事件历史，并在 MQTT 实时事件接入后做一次回填对账。

### 3. 持续同步

- 赛前低频 HTTP 对账；接近计划时间、`global=1`、局间和出现异常时提高到约 5 秒。具体频率应配置化并遵守提供方限制。
- 收到 `csgo-detail-bp`、MQTT 重连、版本断链、全零 `from_ver`、状态矛盾或解析失败时立即触发 HTTP 重同步。
- 普通非零 `from_ver` 维护增量链；`from_ver` 全零包作为独立基线/重置分支幂等合并，不能推进普通链游标。
- MQTT 断线时重新获取短期凭据并重新订阅；使用退避重连，同时依赖 HTTP 保持状态可用。
- 完赛后至少再取两份一致 HTTP 快照。`csgo_mc_2395547` 的后一份 HTTP 将 Mirage `start_time` 从 `1784550794` 修订为 `1784550834`，`end_time` 从 `1784553565` 修订为 `1784553564`，证明首报时间不能冻结。

### 4. 完整事件历史

- 初次请求 `event/log?update_version=0&limit=500`；每页取最旧事件的 `info.update_version` 作为下一页游标。
- 不信任 `not_more`；在空页、短页、游标不再前进或整页已重复时停止。
- 以 `update_version` 为主键去重；击杀等日志可辅以内部 `event_id`。
- MQTT `csgo-event-log` 实时追加；地图结束、重连和完赛时分别做按图和全场回填，消除断线窗口。

### 5. “人肉页面能看到的都要有”的接口组合

- `/data`：页头、赛事/战队、BP、系列赛及每图状态、比分、选手实时状态、统计、地图图标和直播配置。
- `/analysis_v1`：赛前选手、战队、地图、实力、近期和交手分析。
- 两套战队近期比赛接口：页头近期比赛与分析表格来源不同，必须都取。
- 事件历史 + 实时事件主题：页面实时日志和按图历史。
- 社区评分标签/列表接口：有数据时生成评分卡片。
- 聊天室可用性接口：公开未登录页通常没有聊天室；登录态聊天应做可选模块，并将会话/连接材料与公开比赛数据隔离。
- 完赛后的赛事内容/详情接口：集锦、回放、报道和话题；空数组也应作为“已请求但无内容”保存。
- 图片、队徽、选手头像、武器和地图资源 URL 已包含在上述 JSON 中；如果目标是离线复现，再按 URL 单独下载并校验哈希。

### 6. 对外输出的最小可靠字段

每次状态更新至少输出：`match_id`、`phase`、`global_status`、`bout_status_vector`、系列赛比分、三张图的地图名/比分/结果/`played`、`source_version`、`observed_at`、`last_http_sync_at`、`mqtt_connected` 和 `consistency`。只有 `consistency=consistent` 时才对外发布新的确认状态；否则保留上一份已确认状态并标记正在重同步。

分类器的核心逻辑应是穷举上述已验证向量，而不是按比分、时间或日志猜测。对于未列出的向量，唯一安全结果是 `unknown_inconsistent`。
